  The file MUST contain these exact XML-tagged sections in this order:

  1. **`<purpose>`** -- One paragraph stating:
     - Analyzes a scanned repository to produce QA_ANALYSIS.md and TEST_INVENTORY.md
     - Consumes SCAN_MANIFEST.md (from scanner) and CLAUDE.md (QA standards)
     - Produces comprehensive testability report with architecture, risks, targets, and pyramid
     - Produces pyramid-based test case inventory with full spec detail per test case
     - Optionally produces QA_REPO_BLUEPRINT.md for Option 1 (dev-only) workflows
     - Spawned by orchestrator after scanner completes successfully

  2. **`<required_reading>`** -- Files the subagent MUST read before analysis:
     - `SCAN_MANIFEST.md` (path provided by orchestrator in files_to_read -- the scanner's output)
     - `templates/qa-analysis.md` (QA_ANALYSIS output format contract)
     - `templates/test-inventory.md` (TEST_INVENTORY output format contract)
     - `templates/qa-repo-blueprint.md` (QA_REPO_BLUEPRINT format -- produce if Option 1 workflow)
     - `CLAUDE.md` (sections: Testing Pyramid, Test Spec Rules, Naming Conventions, Quality Gates, Module Boundaries)
     - Note: "Read ALL these files BEFORE producing any output. The subagent MUST read CLAUDE.md Test Spec Rules to understand assertion specificity requirements."

  3. **`<process>`** with these `<step>` elements:

     **`<step name="read_inputs" priority="first">`**
     - Read SCAN_MANIFEST.md completely (path from orchestrator's files_to_read)
     - Read templates/qa-analysis.md -- extract 6 required sections and field definitions
     - Read templates/test-inventory.md -- extract 5 required sections and per-test-case mandatory fields
     - Read templates/qa-repo-blueprint.md -- extract 7 required sections
     - Read CLAUDE.md sections: Testing Pyramid (pyramid percentages), Test Spec Rules (every test case mandatory fields), Naming Conventions (test ID format), Quality Gates (assertion specificity rules), Module Boundaries (analyzer reads/produces)
     - Verify SCAN_MANIFEST.md has all 5 sections populated. If incomplete, note gaps for the Assumptions section.

     **`<step name="assumptions_checkpoint">`**
     - Read SCAN_MANIFEST.md completely
     - List 3-8 assumptions about the codebase with evidence from the scan data. Examples:
       "Auth uses JWT based on jsonwebtoken in package.json dependencies"
       "Database is PostgreSQL based on Prisma datasource in schema.prisma"
       "Payment processing uses Stripe based on stripe package in dependencies"
     - List 0-3 questions that genuinely affect analysis quality. Examples:
       "Is the Stripe integration in production or test mode?"
       "Are there additional API endpoints not captured in route files?"
     - Return checkpoint with this exact structure:
     ```
     CHECKPOINT_RETURN:
     completed: "Read SCAN_MANIFEST.md, identified assumptions and questions"
     blocking: "Need user confirmation on assumptions before generating analysis"
     details:
       assumptions:
         - assumption: "[text]"
           evidence: "[what in the code supports this]"
         - ...
       questions:
         - "[question text]"
         - ...
     awaiting: "User confirms assumptions are correct or provides corrections. User answers questions if any."
     ```

     **`<step name="produce_qa_analysis">`**
     - After assumptions confirmed (or if running in auto-advance mode), produce QA_ANALYSIS.md
     - MUST include ALL 6 sections from templates/qa-analysis.md:
       a. Architecture Overview: system_type, language, runtime, framework, database, authentication, integrations, deployment. Entry Points table. Internal Layers diagram.
       b. External Dependencies: table with dependency, purpose, version, risk_level (HIGH/MEDIUM/LOW), justification. Risk classification per CLAUDE.md rules.
       c. Risk Assessment: RISK-NNN IDs, area, severity, description, evidence (cite specific file/function), testing_implication. Every risk MUST cite a specific file or function.
       d. Top 10 Unit Test Targets: rank 1-10 by composite score (business_impact 40% x complexity 30% x change_frequency 30%). Each with module_path, function_or_method, why_high_priority, complexity assessment, suggested_test_count.
       e. API/Contract Test Targets: grouped by resource, each with endpoint, request_contract, response_contract, auth_required, test_priority.
       f. Recommended Testing Pyramid: ASCII visualization, tier table with percentage/count/rationale, justification paragraph specific to this app.
     - Pyramid percentages MUST sum to 100%
     - All risks must reference actual code (no generic "SQL injection is possible" without a specific file)

     **`<step name="produce_test_inventory">`**
     - Produce TEST_INVENTORY.md with ALL 5 sections from templates/test-inventory.md:
       a. Summary: total_tests, counts and percentages per tier, p0/p1/p2 counts, coverage_narrative
       b. Unit Tests (target 60-70%): EVERY test case MUST have ALL 7 mandatory fields: test_id (UT-MODULE-NNN), target (file:function), what_to_validate, concrete_inputs (actual values), mocks_needed, expected_outcome (exact value/error/state), priority (P0/P1/P2)
       c. Integration/Contract Tests (target 10-15%): test_id (INT-MODULE-NNN), components_involved, what_to_validate, setup_required, expected_outcome, priority
       d. API Tests (target 20-25%): test_id (API-RESOURCE-NNN), method_endpoint, request_body (exact JSON), headers, expected_status (exact code), expected_response (key fields), priority
       e. E2E Smoke Tests (target 3-5%, max 3-8 tests): test_id (E2E-FLOW-NNN), user_journey (step-by-step), pages_involved, expected_outcome, priority (always P0)
     - Test count depends on repo size/complexity -- follows pyramid distribution
     - Test IDs MUST be unique across entire document (no duplicates)
     - Summary counts MUST match actual test case counts in each section

     **CRITICAL ANTI-PATTERN CHECK (embed this in the step instructions):**
     Before finalizing TEST_INVENTORY.md, scan EVERY expected_outcome field. If ANY contains these vague words without a concrete value, REWRITE it:
     - "correct" -> specify the exact correct value
     - "proper" -> specify what "proper" means
     - "appropriate" -> specify the exact appropriate behavior
     - "works" -> specify the observable result
     - "valid" -> specify what makes it valid
     Example: "Returns correct data" MUST become "Returns {id: 'usr_123', email: 'test@example.com', role: 'customer'}"

     **`<step name="produce_blueprint">`**
     - Check if orchestrator indicated Option 1 (dev-only) workflow via `workflow_option` parameter in prompt
     - If `workflow_option` is 1 (or not specified -- default to producing it): produce QA_REPO_BLUEPRINT.md with all 7 sections from templates/qa-repo-blueprint.md
     - If `workflow_option` is 2 or 3: skip this step (QA repo already exists)
     - Blueprint includes: project info, folder structure, recommended stack, config files, execution scripts, CI/CD strategy, definition of done

     **`<step name="write_output">`**
     - Write QA_ANALYSIS.md to the output path specified by orchestrator
     - Write TEST_INVENTORY.md to the output path specified by orchestrator
     - If produced, write QA_REPO_BLUEPRINT.md to the output path specified by orchestrator
     - Commit using: `node bin/qaa-tools.cjs commit "qa(analyzer): produce QA_ANALYSIS.md and TEST_INVENTORY.md" --files {output_paths}`
     - If blueprint also produced: include it in commit files

     **`<step name="validate_output">`**
     - Run quality gate checks against produced artifacts
     - Verify QA_ANALYSIS.md: all 6 sections present, pyramid sums to 100%, risks cite specific files
     - Verify TEST_INVENTORY.md: all test IDs unique, all mandatory fields filled, no vague expected outcomes, summary counts match actual counts
     - If SCAN_MANIFEST.md was incomplete: document gaps in QA_ANALYSIS.md assumptions section

  4. **`<output>`** -- What the analyzer produces:
     - QA_ANALYSIS.md at output path (always)
     - TEST_INVENTORY.md at output path (always)
     - QA_REPO_BLUEPRINT.md at output path (Option 1 only)
     - Return to orchestrator: file paths, total test count, pyramid breakdown (unit/integration/api/e2e counts), risk count (high/medium/low)

  5. **`<quality_gate>`** -- MUST embed quality gates from BOTH templates/qa-analysis.md AND templates/test-inventory.md VERBATIM, PLUS these additional analyzer-specific checks:
     - `[ ] No expected outcome uses "correct", "proper", "appropriate", or "works" without concrete value`
     - `[ ] Pyramid percentages sum to 100%`
     - `[ ] Test IDs are unique and follow naming convention (UT-MODULE-NNN, INT-MODULE-NNN, API-RESOURCE-NNN, E2E-FLOW-NNN)`
     - `[ ] Every unit test has all 7 mandatory fields (test_id, target, what_to_validate, concrete_inputs, mocks_needed, expected_outcome, priority)`
     - `[ ] Every risk cites a specific file or function as evidence`
     - `[ ] Summary tier counts match actual test case counts in each section`
     - `[ ] Assumptions section documents any gaps from incomplete SCAN_MANIFEST.md`

  6. **`<success_criteria>`** -- Summary for orchestrator:
     - QA_ANALYSIS.md exists with all 6 required sections
     - TEST_INVENTORY.md exists with all 5 required sections
     - Test case count follows pyramid distribution
     - Return: file paths, total test count, pyramid breakdown, risk count

  **CRITICAL anti-patterns to avoid (from research):**
  - Do NOT embed template content in the agent .md -- agent reads templates at runtime
  - Do NOT hardcode the output file path -- orchestrator passes it in prompt
  - Do NOT produce SCAN_MANIFEST.md -- that is qa-scanner's artifact (module boundary violation)
  - Do NOT skip reading CLAUDE.md -- the test spec rules and quality gates are ESSENTIAL for correct output
  - Do NOT produce vague test cases -- every expected outcome must have a concrete value
