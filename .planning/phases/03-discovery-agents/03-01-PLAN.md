  Create `agents/` directory at project root, then create `agents/qaa-scanner.md`.

  The file MUST contain these exact XML-tagged sections in this order:

  1. **`<purpose>`** -- One paragraph stating:
     - Scans a developer repository to produce SCAN_MANIFEST.md
     - Reads file tree, package manifests, config files, and source code
     - Detects framework, language, runtime, and all testable surfaces
     - Spawned by orchestrator as first stage of QA pipeline
     - Accepts DEV repo path and output path from orchestrator prompt

  2. **`<required_reading>`** -- Files the subagent MUST read before scanning:
     - `templates/scan-manifest.md` (output format contract)
     - `CLAUDE.md` (sections: Framework Detection, Module Boundaries, Verification Commands)
     - Note: "Read these files BEFORE any scanning operation. Do NOT skip."

  3. **`<process>`** with these `<step>` elements:

     **`<step name="read_templates" priority="first">`**
     - Read templates/scan-manifest.md completely
     - Read CLAUDE.md sections: Module Boundaries, Verification Commands, Framework Detection
     - Extract the 5 required sections and their field requirements
     - Extract the quality gate checklist

     **`<step name="detect_project">`**
     - Detection priority order (depth-first by priority):
       a. Package manifests: `package.json`, `requirements.txt`, `setup.py`, `pyproject.toml`, `*.csproj`, `pom.xml`, `build.gradle`, `go.mod`, `Gemfile`, `composer.json`
       b. Configuration files: `tsconfig.json`, `vite.config.*`, `next.config.*`, `nuxt.config.*`, `angular.json`, `vue.config.*`, `webpack.config.*`
       c. Lock files (version pinning): `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`
       d. File extension frequency analysis
       e. Source code patterns: import statements, decorator patterns
     - Include a COMPREHENSIVE framework-to-file-pattern mapping table:
       Node.js/Express: `.ts`, `.js`, `.mjs`
       Python/FastAPI/Django: `.py`
       .NET/ASP.NET: `.cs`, `.razor`, `.cshtml`
       Java/Spring: `.java`, `.kt`
       Go: `.go`
       Ruby/Rails: `.rb`, `.erb`
       PHP/Laravel: `.php`, `.blade.php`
       React/Next.js: `.tsx`, `.jsx`
       Vue/Nuxt: `.vue`, `.ts`
       Angular: `.component.ts`, `.service.ts`, `.module.ts`
     - Populate Project Detection section fields: framework, language, runtime, component_pattern, package_manager (required); build_tool, test_framework_existing, database, css_approach (optional)
     - Assign detection confidence: HIGH (clear detection from package manifest + config), MEDIUM (some ambiguity), LOW (uncertain stack)
     - Record detection sources (which files informed each field)
     - Monorepo handling: check for `workspaces` in package.json, `lerna.json`, `pnpm-workspace.yaml`, `nx.json`. If detected, scan each package as separate unit; produce one manifest with packages listed.

     **If confidence is LOW or framework unknown:**
     Return checkpoint with this exact structure:
     ```
     CHECKPOINT_RETURN:
     completed: "Scanned file tree, read package manifests"
     blocking: "Framework detection uncertain"
     details: "Found files: [list]. Partial detection: [what was found]. Confidence: LOW"
     awaiting: "User confirms framework and language, or provides additional context"
     ```

     **`<step name="build_file_list">`**
     - Use Glob tool to discover all source files
     - Exclude: `node_modules/`, `dist/`, `build/`, `.next/`, `coverage/`, test files (`*.test.*`, `*.spec.*`, `*.stories.*`), config-only files, lockfiles, static assets
     - For each file, determine: file_path, component_name (from filename or default export), type (page/component/service/utility/model/middleware/route/controller/config), interaction_density (HIGH/MEDIUM/LOW), priority_order (integer rank)
     - Priority ordering: forms and interactive components first (HIGH), pages and views second (MEDIUM-HIGH), services with business logic (HIGH), API routes/controllers (MEDIUM-HIGH), static display (LOW-MEDIUM), utilities (LOW-MEDIUM), models/types (LOW)
     - Include line_count and exports_count when available
     - Compute summary statistics: total_files, files_by_type, files_by_priority

     **`<step name="identify_testable_surfaces">`**
     - Categorize all testable entry points into 5 categories:
       a. Pages/Views: route path, component file, description
       b. Forms: form name, component file, fields list, submission endpoint
       c. API Endpoints: method, path, controller/handler file, auth required (yes/no)
       d. Business Logic Modules: module file, key functions, why testable
       e. Middleware: middleware file, what it does, routes it applies to
     - Read source files as needed to extract: API endpoint definitions from route files, form fields from component files, function signatures from service files

     **`<step name="detect_frontend">`**
     - Detect frontend components: framework (React/Vue/Angular/None), component files count, interactive elements count
     - Set `has_frontend: true` if component files found (*.tsx, *.jsx, *.vue, *.component.ts)
     - Set `has_frontend: false` if backend-only
     - This flag is CRITICAL -- orchestrator uses it to decide whether to spawn testid-injector

     **`<step name="decision_gate">`**
     - Apply decision rules from scan-manifest.md template:
       If total files = 0: STOP "No source files found"
       If 0 component files AND frontend expected: STOP "Expected frontend components but found none"
       If backend-only: PROCEED "Skip testid-inject, proceed to analyze"
       If mixed frontend + backend: PROCEED "Full pipeline -- include testid-inject for frontend components"
       If only config/utility: STOP "No testable surfaces detected"
     - Output: decision (PROCEED/STOP), reason, pipeline_note, confidence (HIGH/MEDIUM/LOW)
     - MUST include `has_frontend: true/false` in decision gate output

     **If decision is STOP and no testable surfaces:**
     Return checkpoint with this exact structure:
     ```
     CHECKPOINT_RETURN:
     completed: "Scanned repo, built file list"
     blocking: "No testable surfaces found"
     details: "Total files: N. Types found: [list]. No services, controllers, or components detected."
     awaiting: "User confirms repo path is correct or provides guidance on testable areas"
     ```

     **`<step name="write_output">`**
     - Write SCAN_MANIFEST.md to the output path specified by orchestrator in the prompt
     - Format must match templates/scan-manifest.md exactly: 5 sections in order (Project Detection, File List, Summary Statistics, Testable Surfaces, Decision Gate)
     - Commit using: `node bin/qaa-tools.cjs commit "qa(scanner): produce SCAN_MANIFEST.md for {project_name}" --files {output_path}`

  4. **`<output>`** -- What the scanner produces:
     - SCAN_MANIFEST.md at the output path specified by orchestrator
     - Return to orchestrator: file path, decision (PROCEED/STOP), has_frontend (true/false), detection_confidence (HIGH/MEDIUM/LOW)

  5. **`<quality_gate>`** -- MUST embed the quality gate from templates/scan-manifest.md VERBATIM (all 10 items), PLUS these additional checks:
     - `[ ] has_frontend field present in Decision Gate (true/false)`
     - `[ ] detection_confidence field present in Decision Gate (HIGH/MEDIUM/LOW)`
     - `[ ] Framework-to-file-pattern mapping covers all 10+ stacks listed in required_reading`
     - `[ ] Output path matches what orchestrator specified (not hardcoded)`

  6. **`<success_criteria>`** -- Summary for orchestrator:
     - SCAN_MANIFEST.md exists at output path
     - All 5 required sections populated
     - Decision Gate is PROCEED or STOP with reason
     - Return: file path, decision, has_frontend flag, detection confidence

  **CRITICAL anti-patterns to avoid (from research):**
  - Do NOT embed template content in the agent .md -- agent reads template at runtime
  - Do NOT hardcode the output file path -- orchestrator passes it in prompt
  - Do NOT assume agent can call other agents -- orchestrator mediates all handoffs
  - Do NOT skip the read-before-write rule -- first step MUST read templates and CLAUDE.md
