Read data-testid-SKILL.md for full naming rules. Read templates/testid-audit-report.md for output format.
    2. `phase_1_scan` -- Detect frontend framework from package.json and file extensions: React (*.jsx, *.tsx), Vue (*.vue), Angular (*.component.html, *.component.ts), Svelte (*.svelte), plain HTML (*.html). List ALL component files, excluding: test files (*.test.*, *.spec.*), story files (*.stories.*), type definitions (*.d.ts). Sort by interaction density: forms first, pages second, layouts third. If no component files found: CHECKPOINT_RETURN with "No frontend component files found despite has_frontend=true".
    3. `phase_2_audit` -- For each component file:
       - Read the file source
       - Identify ALL interactive elements: <button>, <input>, <select>, <textarea>, <a> (links), <form>, <img> (when dynamic), <table>, modals/dialogs, dropdowns, toggles, checkboxes, radios
       - For each element: record line number, element tag and type, current selector (data-testid value, className, name attribute, or "none")
       - Classify priority: P0 (form inputs, submit buttons, primary actions), P1 (navigation links, secondary buttons, error/alert messages, toggles), P2 (decorative images, badges, containers with dynamic content)
       - For elements WITH existing data-testid: mark as "EXISTING -- no change", do NOT modify
       - For elements WITHOUT data-testid: propose a value following {context}-{description}-{element-type} convention from data-testid-SKILL.md
       - Calculate coverage score: existing_testid_count / total_interactive_elements * 100
       - Apply decision gate thresholds: >90% SELECTIVE, 50-90% TARGETED, 1-49% FULL PASS, 0% P0 FIRST
       - Produce TESTID_AUDIT_REPORT.md at orchestrator-specified path

    4. `audit_checkpoint` -- CHECKPOINT_RETURN with audit results for user review:
       ```
       CHECKPOINT_RETURN:
       completed: "Scanned {N} files, found {M} interactive elements, {X} missing data-testid"
       blocking: "Need user approval before injecting data-testid attributes"
       details:
         coverage_score: "{X}%"
         decision: "{SELECTIVE|TARGETED|FULL PASS|P0 FIRST}"
         p0_missing: {count}
         p1_missing: {count}
         p2_missing: {count}
         non_compliant_existing: {count}
         report_path: "{path to TESTID_AUDIT_REPORT.md}"
       awaiting: "User reviews proposed values in TESTID_AUDIT_REPORT.md and approves. User may reject individual elements or rename proposals."
       ```
       Per CONTEXT.MD: "Audit-first workflow: Phase 1 produces TESTID_AUDIT_REPORT with proposed values -> user reviews -> Phase 2 injects only approved items."
       If running in auto-advance mode: proceed with P0 defaults (inject P0 elements only).

    5. `phase_3_inject` -- Per CONTEXT.MD: "Injects on a separate branch: qa/testid-inject-{date}. Working copy stays clean."

       - For each APPROVED element (from user review or P0 defaults in auto-advance):
         - Add data-testid as the LAST attribute before the closing >
         - Framework-specific injection:
           - JSX/TSX: data-testid="value" as JSX attribute
           - Vue: data-testid="value" as HTML attribute in template
           - Angular: attr.data-testid="value" or data-testid="value" in template
           - HTML: data-testid="value" as HTML attribute
         - Dynamic list items: use template literals: data-testid={`context-${item.id}-element-type`}
         - Third-party components: (a) props passthrough if supported, (b) wrapper div if not, (c) inputProps/slotProps for MUI
       - Per CONTEXT.MD: "Existing data-testid values: preserved as-is. Non-compliant existing IDs reported in audit with offer to rename (user decides per ID)."
       - Per CONTEXT.MD: "Default: inject P0 elements only (buttons, inputs, forms, links, modals). P1-P2 offered as optional follow-up."
       - Preserve all existing formatting -- change NOTHING except adding the data-testid attribute

    6. `phase_4_validate` -- Validate all modified files:
       - Syntax check: run the appropriate linter/compiler on each modified file
       - Uniqueness check: no duplicate data-testid values within the same page/route scope
       - Convention compliance: all injected values follow {context}-{description}-{element-type} pattern
       - Non-interference check: diff each file, verify ONLY data-testid additions (no other code changes)
       - If any validation fails: revert the specific file and report the issue

    7. `produce_report` -- Write/update TESTID_AUDIT_REPORT.md with final results (post-injection). Write INJECTION_CHANGELOG.md documenting every injection: file, line, element, proposed value, action (injected/skipped/deferred). Commit on the injection branch: node bin/qaa-tools.cjs commit "qa(testid-injector): inject {N} data-testid attributes across {M} components" --files {modified_files} {report_path} {changelog_path}

       ```
         coverage_after: {Y}%
         elements_injected: {count}
         files_modified: {count}
         non_compliant_reported: {count}
         commit_hash: "{hash}"
       ```
