**Required Sections (in this order):**

1. **Title + Purpose block** -- "TEST_INVENTORY.md Template", purpose: complete pyramid-based test case inventory, producer: qa-analyzer, consumers: qa-planner + qa-executor.

2. **Section: Summary** -- Total test count, breakdown by pyramid tier (count + percentage), coverage assessment narrative, priority distribution (P0/P1/P2 counts).

3. **Section: Unit Tests (60-70%)** -- Per test case, ALL of these fields are MANDATORY:
   - Test ID: `UT-MODULE-NNN` format
   - Target: `file_path:function_name`
   - What to validate: one-sentence description
   - Concrete inputs: actual values, not "valid data"
   - Mocks needed: list or "None (pure function)"
   - Explicit expected outcome: exact return value, exact error, exact state change
   - Priority: P0/P1/P2

   Anti-pattern callout: "Returns correct value" is NOT an expected outcome. "Returns 239.47" IS.

4. **Section: Integration/Contract Tests (10-15%)** -- Per test:
   - Test ID: `INT-MODULE-NNN`
   - Components involved: which modules interact
   - What to validate: the interaction contract
   - Setup required: database state, mock services
   - Expected outcome: specific behavior when components interact
   - Priority: P0/P1/P2

5. **Section: API Tests (20-25%)** -- Per test:
   - Test ID: `API-RESOURCE-NNN`
   - Method + Endpoint: `POST /api/v1/orders`
   - Request body/params: exact JSON payload
   - Headers: auth token requirements
   - Expected status code: exact number (200, 201, 400, 401, 404)
   - Expected response shape: key fields with types/values
   - Priority: P0/P1/P2

6. **Section: E2E Smoke Tests (3-5%)** -- Max 3-8 critical paths:
   - Test ID: `E2E-FLOW-NNN`
   - User journey: step-by-step description
   - Pages involved: list of views/routes
   - Expected outcome: final state user observes
   - Priority: all P0

**Guidelines section** with DO/DON'T:
- DO: Use concrete values in every input and expected outcome field

- DON'T: Write "returns correct data" -- specify the exact data
